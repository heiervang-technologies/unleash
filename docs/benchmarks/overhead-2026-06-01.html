Conclusions

Claude is the only agent that pays a real startup tax. Its wrapper path adds +130 ms of wall clock and +53.0 MB of peak RSS — large enough to see on every launch. Every other agent's overhead is within noise.
Root cause (confirmed by 2×2 factorial against claude --version): Claude's heavy startup path is triggered when either --dangerously-skip-permissions or any --plugin-dir flag is present. Each alone costs the same fixed ~+100 ms / ~+54 MB; passing both is not additive. Plugin count doesn't matter either — 9 --plugin-dir args cost the same as 1. Unleash adds both flag families unconditionally for claude (see src/launcher.rs::run_agent), so every wrapped launch lands in the heavy path.

B  claude --dangerously-skip-permissions --version                  ~170 ms / 266 MB

D  claude --dangerously-skip-permissions --plugin-dir DIR --version ~170 ms / 266 MB

Speed (wall + CPU)

  • Median wall-clock overhead across all 7 agents is +10 ms (mean +26 ms). Pulled almost entirely by claude (+130 ms); the other six are 0–20 ms, which is at or below GNU time's 10 ms resolution floor.
  • Near-zero wall overhead: codex, agy, gemini, opencode, pi, hermes. These agents are dominated by their own startup (node + JS parse, Python import) so the Rust wrapper's ~10–20 ms exec/dispatch cost vanishes in the noise.
  • CPU overhead tracks wall closely. claude's wrapped path burns +150 ms more CPU — actual JS parsing / module-init work, not sleep. The factorial above pins this to claude's plugin/permissions subsystem, not unleash's own pre-exec code.

Memory (peak RSS)

  • Median RSS overhead: +0 KB (mean +7.3 MB). Again driven by claude (+53.0 MB).
  • 6 of 7 agents land within ±2 MB of their direct baseline: codex, agy, gemini, opencode, pi, hermes. Several show small negative deltas (opencode, agy, pi) — those are scheduler / pagecache noise, not real savings.
  • The +53 MB on claude is paid by claude itself when it enters the plugin/permissions path. The Rust wrapper holds ~10 MB; everything beyond that is extra JS modules and V8 heap inside the child process. The wrapper can't shrink it directly — only avoid triggering the path.

Optimization candidates (claude)

  • Short-circuit meta commands. Detect --version, --help, --print, doctor etc. and exec claude clean — no --dangerously-skip-permissions, no --plugin-dir. Recovers the full ~100 ms / ~54 MB for short-lived invocations (including every hook subprocess that calls back into unleash).
  • Consolidate plugins. 1 plugin-dir costs the same as 9, so merging the bundled plugins into a single meta-plugin doesn't help the per-launch fixed cost — but it does shrink the settings.json hook block and cut per-hook dispatch latency at runtime. Worth doing for hot-path hooks (Stop, PreCompact).
  • Skip hook sync on no-op launches. HookManager::sync_plugin_hooks writes settings.json on every unleash claude. Hash the plugin set and skip the write when unchanged — saves a small but consistent disk hit plus a few ms.
  • Drop the telemetry-dir scrub. rm -rf ~/.claude/telemetry/ && mkdir runs on every launch (launcher.rs:526). Move to a daily cron / one-shot, or only run when the dir is >N MB.
  • Upstream ask. Claude's plugin/permissions init could be lazy — only load when permissions are actually queried or a hook fires. That's a Claude Code change, not unleash's, but worth filing.

Bottom line

  • For 6 of 7 agents the wrapper is effectively free — under 25 ms wall, under 2 MB peak RSS. That's well below what users will perceive at launch.
  • Claude's overhead lives inside claude, not unleash. Whatever turns on --dangerously-skip-permissions or --plugin-dir triggers a ~+100 ms / ~+54 MB plugin/permissions init. The wrapper can avoid this for meta commands but can't shrink it for real sessions without an upstream change.
  • This is a startup measurement — it doesn't model overhead during a live session (hooks, plugin dispatch, supercompact). A future suite should cover that with a synthetic turn-loop.
